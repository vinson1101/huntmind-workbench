# TalentFlow 招聘审核系统 — 完整流程说明

> 版本：V1.4
> 最后更新：2026-03-24
> 负责人：HuntMind（AI 招聘决策 Agent）

---

## 一、系统定位

**TalentFlow** 是 HuntMind 的招聘执行后端，负责：
1. 简历读取与解析
2. 调用 HuntMind 进行 AI 评估决策
3. 生成结构化报告
4. 将结果写入飞书多维表格（HR 操作界面）

**一轮 skill = 一轮完整招聘处理**，一轮的结果落一张 Runs 记录 + N 条 Candidates 记录。

---

## 二、整体流程

```
阶段一：HR 准备
├── 1. 准备 JD（提供原始文本或结构化 jd.json）
└── 2. 获取简历（两种方式）

阶段二：AI 处理
├── 3. 简历保存到 inbox
├── 4. 简历解析读入
├── 5. HuntMind AI 评估决策
└── 6. Pipeline 校验与报告生成

阶段三：结果输出
├── 7. 写入飞书多维表格（Runs 表 + Candidates 表）
└── 8. HR 在飞书中审核、联系候选人、填写反馈
```

---

## 三、阶段详解

### 阶段一：HR 准备（人工）

#### 1.1 准备 JD

HR 提供 JD，有两种方式：

**方式A — 直接提供结构化 jd.json**（推荐）：
```json
{
  "title": "高级产品经理",
  "must_have": ["5年以上B端产品经验", "从0到1产品搭建经验"],
  "nice_to_have": ["AI/硬件行业背景", "出海经验"],
  "salary_range": "25-40K",
  "location": "深圳南山",
  "company_context": "深圳某AI机器人创业公司，商用清洁机器人赛道",
  "priority_focus": "skill"
}
```

**方式B — 提供自然语言 JD 描述**：
AI 自动解析为结构化 JSON（增强能力，后续迭代）。

#### 1.2 获取简历（两种方式）

**方式A — 飞书云盘**
- HR 将候选人简历 PDF 上传到飞书云盘「简历」文件夹
- 告知 AI 该文件夹的 `folder_token`
- AI 调用 `load_feishu_resume_files()` 下载，汇入 `inbox/`

**方式B — 聊天窗口直接发**
- HR 在飞书私信里直接发送 PDF 附件（单文件或多文件）
- AI 接收附件，保存到 `inbox/`

> 两种方式可以混合使用，同一批次里部分来自飞书云盘、部分来自聊天窗口，最终都进入 `inbox/` 统一处理。

---

### 阶段二：AI 处理（自动）

#### 2.1 简历保存到 inbox

两种来源的处理差异仅在于获取路径不同，主处理逻辑**不分叉**。

`source` 元信息（platform / file_id / file_name）**必须保留**，用于：
- 追溯简历原始来源
- 生成 Candidates 表的「来源平台」和「简历链接」字段

| 来源 | 处理方式 | source.platform | source.file_id | resume_link |
|------|---------|----------------|----------------|-------------|
| 飞书云盘 | 调用 `load_feishu_resume_files()` 下载 | `"feishu"` | 飞书 file_token | 指向飞书文件 |
| 聊天窗口附件 | 接收附件保存到 inbox | `"local"` | 无 | 空字符串 |

#### 2.2 简历解析读入

`resume_ingest.py` 扫描 `inbox/`，解析所有 PDF，生成 `batch_input.json`（包含候选人 ID、姓名、简历文本、来源信息）。

#### 2.3 HuntMind AI 评估决策

**7维评分体系**：

| 维度 | 权重 | 评估内容 |
|------|------|---------|
| 硬技能匹配 | 30% | 与 JD 硬技能要求的匹配程度 |
| 经验深度 | 25% | 相关领域工作年限 + 复杂度 |
| 创新潜能 | 15% | 项目成果、创新意识、学习能力 |
| 目标拆解执行 | 15% | 目标设定、拆解、执行力 |
| 团队融合 | 5% | 跨部门协作、沟通风格 |
| 意愿度 | 5% | 对岗位/薪资/地点的接受度 |
| 稳定性 | 5% | 离职原因、换工作频率、在职状态 |

**decision 决策标准**：
- `strong_yes`：核心技能完全匹配，建议立即联系
- `yes`：大部分匹配，值得联系
- `maybe`：部分匹配，作为备选
- `no`：核心不匹配，不建议联系

**核心原则：排名必须有，推荐不必须有。**
- 每批都按分数降序排名（1=最高分）
- 不是每批都必须有 strong_yes 或 A 类——整批都不达标也是正常结果
- 禁止为了"有输出"而强行凑推荐理由

#### 2.4 Pipeline 校验与报告生成

1. **schema 校验**：`validate_model_output.py` 检查输出格式
2. **Sanitize**：清洗 Markdown 格式、截断超长文本
3. **加权计算**：按模板权重重算总分，覆盖模型原始分数
4. **质量门禁**：检测 identity_conflict、low_variance、weak_ranking 等异常
5. **报告生成**：`final_output.json` + `owner_summary.md`

---

### 阶段三：结果输出

#### 3.1 写入飞书多维表格

自动写入两张表（配置从环境变量或配置文件读取，不写在文档里）：

**Runs 表**（每批次一条）：
- 批次ID、JD信息、质量分、Top候选人名单、摘要

**Candidates 表**（每候选人一条）：
- 基础信息（姓名、ID、来源、简历链接）
- 决策结论（decision、priority、score、action_timing）
- 行动字段（推荐理由、风险提示、话术模板）
- 7维评分 + 证据摘要
- 身份冲突标记

**简历链接写入规则**：

| 来源 | source 写入 | 简历链接 |
|------|------------|---------|
| 飞书云盘 | `file_id: "UL4rbKoR5oXSo..."` | `https://...feishu.cn/file/{file_id}` |
| 聊天窗口 | `file_name: "张三.pdf"` | 空字符串（`""`） |

#### 3.2 飞书 Candidates 视图使用建议

> ⚠️ 以下为推荐默认值，非硬约束。字段展示顺序和排序策略允许在实战中人工微调。

**推荐默认视图**：
- 过滤条件：`批次ID` = 当前 run_id
- 排序：`排名` 升序（同 rank 时按综合评分降序）
- 默认使用该视图，不建议直接浏览全量 Candidates 总表

**推荐默认字段顺序**：
排名 → 候选人姓名 → 简历链接 → 角色标签 → 决策结论 → 优先级 → 行动时机 → 综合评分 → 核心判断 → 风险提示 → 核实问题 → 跟进状态 → 面试结果 → 最终结果

> 底层写表 schema 应保持稳定。视图 UX（字段顺序、排序、筛选条件）属于使用层，允许在实战中迭代调整，不需要频繁改动底层写表逻辑。

## 四、字段追踪

### 简历来源 → resume_link 链路

```
飞书云盘简历
    ↓ HR 告知 folder_token
load_feishu_resume_files(folder_token)
    ↓ 下载到 tmp/feishu_resumes/
    ↓ 记录 file_id = "UL4rbKoR5oXSo..."
    ↓ 写入 batch_input.candidates[i].source.file_id
    ↓
feishu_bitable_writer.py
    ↓ 读取 file_id
    ↓ 生成 resume_link = "https://...feishu.cn/file/{file_id}"
    ↓ 写入「简历链接」列
```

```
聊天窗口简历
    ↓ HR 直接发送附件
保存到 inbox/ → 本地路径
    ↓ source.platform = "local"，无 file_id
    ↓ resume_link = ""（空）
```

## 七、关键原则

1. **inbox 汇流**：所有简历最终进入 `inbox/`，主处理逻辑不因来源分叉
2. **source 元信息必须保留**：`source` 字段用于追溯和 resume_link 生成，不得丢弃
3. **幂等写入**：同一批次重复发布不会丢失数据，HR 反馈字段不会被覆盖
4. **排名必须有，推荐不必须有**：整批都不达标是正常结果，不是系统失败
5. **写表 schema 稳定，视图 UX 可迭代**：底层字段结构不变，视图层允许实战微调
